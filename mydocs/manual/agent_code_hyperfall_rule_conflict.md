# Claude와 Codex 기본 동작과 하이퍼-워터폴 충돌 규칙

## 목적

Claude와 Codex는 기본적으로 빠른 실행과 자율 수정에 최적화되어 있다. 이 프로젝트는 하이퍼-워터폴 방법론을 적용하므로, 기본 동작이 프로세스 규율과 충돌할 수 있다. 이 문서는 충돌 시 우선순위와 처리 기준을 정의한다.

## 충돌 지점

1. 즉시 구현 성향 vs 단계 승인 게이트
2. 누락 문서 보완 없이 코드 우선 수정
3. 타스크 종료/이슈 클로즈의 성급한 처리
4. 작업 범위 확장(관련 리팩터링, 불필요 생성물 갱신)

## 우선순위 규칙

1. 작업지시자 명시 지시
2. `AGENTS.md` 강제 규칙
3. task 문서(`plans`, `working`, `report`, `orders`)
4. 자동화 도구/에이전트 기본 동작

상위 규칙과 충돌하면 하위 규칙을 적용하지 않는다.

## 운영 규칙

- 기본 원칙은 `이슈 -> 브랜치 -> 오늘할일 -> 계획서 -> 구현 -> 검증 -> 최종 보고서 -> PR` 순서다.
- 각 단계 완료 후 다음 단계로 넘어가기 전에 승인 여부를 확인한다.
- 같은 스레드에서 작업지시자가 "계속 진행", "다음 단계 진행"처럼 명시 지시하면 해당 단계 승인으로 간주한다.
- 이슈 close는 작업지시자 승인 또는 PR merge 이후에만 수행한다.
- 문서 수정은 기존 내용을 먼저 읽고 필요한 부분만 수정하며, 불가피할 때만 내용을 추가한다.
- 문서 위치/파일명 규칙을 위반한 산출물과 다음 작업에 필요하지 않은 로컬/원격 부산물은 즉시 정리한다.

## 실무 체크리스트

- 수정 전: 현재 이슈 번호, 브랜치, 계획서 존재 여부 확인
- 수정 중: 변경 범위가 승인된 요구와 일치하는지 점검
- 수정 후: 검증 실행, 보고서 갱신, PR 본문에 결과 반영, merge 후 부산물 정리

## 예외 처리

- 작업지시자가 단순 문구 수정, 오탈자 수정, 조사 요청만 지시한 경우 절차를 필요한 수준으로 축소할 수 있다.
- 단, GitHub Issue가 이미 열린 작업은 최소한 최종 보고서와 PR 본문에 결과를 남긴다.
