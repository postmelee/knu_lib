# Task 000 — Hyper-Waterfall 기준선 도입 계획

> GitHub Issue 생성 전 기준 작업이다. Issue가 생성되면 파일명을 실제 번호 기준으로 변경하거나 후속 계획서에서 참조한다.

## 목표

KNU Library 저장소에 `mydocs` 기반 Hyper-Waterfall 운영 체계를 도입하고, 현재 코드/문서 상태를 다음 개발 사이클의 기준선으로 고정한다.

## 범위

- `mydocs` 디렉토리 구조 생성
- 일일 작업판 생성
- KNU Library용 Hyper-Waterfall 운영 규칙 작성
- 현재 코드와 기존 `docs` 자료의 상태 차이 기록
- 다음 마일스톤과 타스크 후보 작성

## 제외

- 기존 `docs/` 파일 이동 또는 삭제
- 앱 기능 코드 변경
- GitHub Issue 생성, 브랜치 변경, 커밋 생성
- 외부 서버 API 호출 또는 실제 로그인 검증

## 실행 단계

1. 현재 저장소의 문서와 코드 구조를 확인한다.
2. `mydocs/orders/20260429.md`를 생성해 오늘 작업을 기록한다.
3. `mydocs/manual/`에 운영 규칙을 작성한다.
4. `mydocs/report/`에 현재 상태 기준선을 작성한다.
5. `mydocs/tech/`에 기존 `docs` 자료의 이관/참조 맵을 작성한다.
6. `mydocs/working/`에 최종 결과 보고서를 작성한다.

## 검증 기준

- `mydocs/orders/20260429.md`가 표준 일일 작업판 형식을 따른다.
- `mydocs` 문서만 추가되고 기존 코드와 `docs` 원본은 변경되지 않는다.
- 현재 구현 상태와 기존 문서의 불일치가 기준선 문서에 명시된다.
- 다음 개발 작업을 Issue 단위로 전환할 수 있는 후보 목록이 존재한다.
